Today's code is tomorrow's technical debt

LLMs are that junior developer who copy-pastas from Stack Overflow without reading, running or understanding the code.

If you're happy to let that developer write the most important parts of your functionality, then fire away! Sometimes, that's okay.

Tips

  • Trust, but verify (dear lord, verify)
  • Check your edge cases
  • Use them to explain code, rather than write it
  • If it sounds confident, its probably wrong
  • It may hallucinate functions, libraries, and entire programming languages

LLMs are solid for explaining code, finding/fixing very acute bugs, and focusing on small tasks like optimization (sometimes). But to write a real app (not a snake game, and nothing that I couldn't write myself in less than 2 hours), they are seriously a pain. It's much more frustrating to get into an argument with Claude because it insists that printing a 5000 line data frame to the terminal is a must if I want "robust" code.

Be dilligent

  • Confidence is not competence
  • Hallucinations
  • Overcomplicated
  • Inconsistent naming and styling
  • Framework confusion
  • Documentation overkill
  • Edge case oversight
  • User base effects

These are the things that you have to handle as a user of LLMs even for simple functions.

Its not always easier to inherit a code base and having to try to understand it than just writing it yourself. The mental overhead of ensuring it is correct, handling the errors, and just making it work can be a lot worse than just writing the code.
